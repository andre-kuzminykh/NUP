Legacy Is Not Necessarily Old

Legacy is any production code the business depends on and the team is afraid to change

Legacy = Business-critical production code + Fear of change

1. Not about age

It can run on a modern stack and still be legacy

2. Key symptom is fear

The real signal is not "old code" — it is "we are afraid to touch it"

3. Appears fast

Code written yesterday becomes legacy if shipped without boundaries/tests

Hidden dependencies
Undocumented logic
Incomplete tests
Fragile deploys

Legacy is not a historical category — it is a system state where change becomes risky

After Analysis, Choose the Mode of Change

Do not code first. Choose the strategy first.

Mode 1: Integration

  • keep the current architecture
  • avoid breaking core contracts
  • add features locally
Best when: business risk is high, SLA is strict, coupling is high, the system still "holds".

Mode 2: Rebuild / Strangler

  • recover product truth
  • design a To-Be model
  • rebuild or replace gradually
Best when: architecture is a big ball of mud, velocity is collapsing, org maturity is strong.
Risk Arch Quality Test Coverage Velocity
Unconscious partial rewrite is the most dangerous path

The Formula of Change

A good spec defines both the delta and the preserved behavior.

What Changes

  • > new behavior
  • > changed validation
  • > new field / added step
  • > changed outcome

What Stays the Same

  • > neighboring scenarios
  • > stable API contracts
  • > preserved side effects
  • > existing edge cases
Regression Zones
Payments Calculations Auth Integrations

A change is safe only when both the delta and the preserved system are explicit.

Build Architecture as As-Is / To-Be / Diff

Architecture is defined by the difference between current and target state.

As-Is

  • - current architecture
  • - real data flows
  • - real constraints
  • - legacy included

To-Be

  • - arch after the change
  • - necessary target state
  • - tied to change request
  • - not a "beauty redesign"

Diff

Delta Layer
  • + what to add
  • ~ what to modify
  • - what to remove
  • = what to keep
Diff defines the real scope of work

Without Diff, there is no controlled development.

1. Validates Reconstruction

2. Reviews Diff / Code

  • - system impact
  • - hidden dependencies
  • - duplicate logic risk

3. Approves Release

  • - risks understood
  • - rollback exists
  • - final responsibility
AI helps explain and generate.

A Prompt Is Not a Security Boundary

Intent written in text is not enforcement. Real safety comes from architecture.

Prompt

Ignorable "Please do not drop the database."
  • guidance
  • preference
  • instruction

Architecture

DB User 'agent' has SELECT only.
  • policy engines
  • tool boundaries
  • execution scopes
  • whitelisted APIs
"Do not do that" "Cannot do that" is protection.

After Release, Fix the New Baseline

A successful change is complete only when product truth, architecture, and regression baseline are updated.

Updated Arch

Updated Tests

New tests become normal protection. Baseline for next iteration.
